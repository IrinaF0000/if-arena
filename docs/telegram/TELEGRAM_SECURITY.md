# Telegram Security Requirements

## 1. Trust model

Telegram Mini App frontend is untrusted. The backend may only trust Telegram identity after validating raw `initData`.

Rules:

- Do not trust `initDataUnsafe`.
- Do not trust frontend-provided user id, username, photo, premium flag, language, or start parameter.
- Do not put the bot token into frontend code.
- Do not log raw `initData`.
- Do not log session tokens.
- Do not expose internal auth failure details.

## 2. Required backend validation

For Telegram auth requests:

1. Check message size before parsing.
2. Extract raw `initData`.
3. Validate Telegram signature/hash according to Telegram documentation.
4. Check `auth_date` freshness.
5. Apply replay/duplicate checks where practical.
6. Extract Telegram user only after validation.
7. Map Telegram user to internal `PlayerId`.
8. Issue internal session identity.

Invalid validation must fail closed.

Current backend helper:

- `src/battle_backend/security/TelegramAuth.hpp` provides `TelegramAuthValidator`.
- The bot token is supplied through `TelegramAuthConfig`; it must come from secure runtime configuration, never frontend code or committed files.
- Validation checks raw init data size, required `hash`, required and fresh `auth_date`, and Telegram HMAC-SHA256 signature.
- `redactTelegramInitDataForLog()` returns generic redaction text for auth logs.
- Replay/duplicate protection is not implemented yet and remains required before production auth.

## 3. Transport requirements

Public Mini App traffic must use:

```text
HTTPS for frontend
WSS for WebSocket backend endpoint
```

Development mode may use localhost exceptions only.

## 4. Session requirements

- Handshake timeout.
- Idle timeout.
- Max unauthenticated sessions.
- Max authenticated sessions.
- Max reconnect attempts per time window.
- Rate limit before and after authentication.
- Session ids must be generated by backend.
- Session ids must not be predictable.

## 5. Input validation

Every WebSocket message must pass the same protocol validation as TCP messages.

Current local WebSocket transport behavior:

- rejects oversized messages before protocol parsing;
- parses inbound text through the shared protocol envelope parser;
- validates message order by session phase before backend handling;
- fails closed on malformed, unknown, oversized, or out-of-order messages;
- provides explicit handshake and idle timeout checks;
- performs the local HTTP Upgrade path used by the Telegram Mini App development build;
- does not trust or extract Telegram identity in the transport layer;
- leaves Telegram `initData` validation to backend-owned auth code before authentication is marked complete.

Reject:

- oversized message;
- malformed JSON;
- unknown message type;
- missing required fields;
- invalid field types;
- command before authentication;
- command before match join;
- command exceeding rate limit;
- command for another player;
- state mutation claims from client.

## 6. Frontend security

Frontend must:

- use strict TypeScript;
- avoid `any`;
- treat inbound WebSocket payloads as unknown;
- validate/guard incoming message shape before using it;
- not use `innerHTML` with untrusted data;
- not store secrets;
- not store long-lived auth data in localStorage unless explicitly reviewed;
- show generic errors to users.

## 7. Logs

Redact:

- raw `initData`;
- Telegram hash;
- session tokens;
- raw auth payload;
- IPs if public report does not need them;
- oversized payloads.

Safe log example:

```text
auth failed: mode=telegram reason=invalid_hash session=s_123
```

Unsafe log example:

```text
auth failed: initData=query_id=...&hash=...
```

## 8. Required tests

- invalid hash;
- stale `auth_date`;
- missing user;
- oversized init data;
- malformed init data;
- replayed auth attempt if replay protection is implemented;
- command before auth;
- command after auth but before join;
- spam commands;
- slow WebSocket client;
- frontend handles backend error without crashing.
